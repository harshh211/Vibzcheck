
## Folder philosophy

- **models** — Pure data, no logic. Each has `fromFirestore()` and `toMap()`.
- **services** — All side effects (network, Firebase). Tested in isolation.
- **providers** — Wire services to UI; hold state. No business logic in screens.
- **screens** — Thin. Read from providers, dispatch actions. No Firebase calls.
- **widgets** — Dumb; take props, emit callbacks.

This separation is deliberate — it's the answer to the curated question *"Choose one screen that became too large and explain how you decomposed responsibilities."* You'll be able to say: *"I kept screens thin by pushing all data access into services and state into providers. For example, SessionScreen only reads sessionProvider; the Firestore transaction for voting lives in FirestoreService."*
